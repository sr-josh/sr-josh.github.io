서버 보안의 문제점 및 필요성

🔴 기존 보안의 문제점

  • M사에 대한 웹 애플리케이션 공격 (SQL Injection, XSS 등), 자동화된 봇(Bot) 공격 증가
  • 공격 패턴 분석 및 대응의 어려움
  • 에러가 발생하는 경우에 한해 담당자가 일일이 검색해서 파이오링크에 요청하거나, iptable에 추가

✅ 보안 강화 필요성

  • 다층 방어 체계 구축 (Defense in Depth)
  • 실시간 공격 탐지 및 차단
  • 자동화된 보안 대응
  • 공격 로그 수집 및 분석
📊 통계: 웹 애플리케이션 공격은 전체 사이버 공격의 약 43%를 차지하며, SQL Injection과 XSS가 가장 흔한 공격 유형입니다.

ModSecurity와 Fail2ban의 역할

🛡️ ModSecurity

웹 애플리케이션 방화벽 (WAF)

주요 기능:

  • HTTP 트래픽 실시간 모니터링
  • SQL Injection 차단
  • XSS (Cross-Site Scripting) 방어
  • 파일 업로드 공격 차단
  • OWASP Top 10 보호

작동 방식:

  • Apache/Nginx 모듈로 동작
  • 규칙 기반 필터링
  • 요청/응답 검사

🚫 Fail2ban

침입 방지 시스템 (IPS)

주요 기능:

  • 로그 파일 모니터링
  • 반복적인 실패 시도 감지
  • IP 기반 자동 차단
  • 서비스별 독립적 보호

작동 방식:

  • 정규식 기반 로그 분석
  • iptables/firewalld 연동
  • 시간 기반 자동 차단 해제
🔗 상호 보완: ModSecurity는 애플리케이션 레벨에서, Fail2ban은 네트워크 레벨에서 보안을 담당하여 완벽한 다층 방어 체계를 구축합니다.

ModSecurity 설치 및 설정

1️⃣ Apache에 ModSecurity 설치

# Ubuntu/Debian sudo apt update sudo apt install libapache2-mod-security2 # CentOS/RHEL sudo yum install mod_security

2️⃣ OWASP Core Rule Set (CRS) 설치

yum install mod_security_crs

3️⃣ 기본 설정

#/etc/httpd/conf.d/mod_security.conf SecRuleEngine On #DetectionOnly SecAuditLog /var/log/httpd/modsec_audit.log
6

Fail2ban 설치 및 설정

1️⃣ Fail2ban 설치

# Ubuntu/Debian sudo apt update sudo apt install fail2ban # CentOS/RHEL sudo yum install epel-release sudo yum install fail2ban # 서비스 시작 sudo systemctl start fail2ban sudo systemctl enable fail2ban

2️⃣ SSH 보호 설정 (/etc/fail2ban/jail.local)

[DEFAULT] bantime = 3600 findtime = 600 maxretry = 5 [sshd] enabled = true port = ssh logpath = /var/log/auth.log maxretry = 3 bantime = 7200

3️⃣ Apache 보호 설정

[apache-auth] enabled = true port = http,https logpath = /var/log/apache2/error.log [apache-modsecurity] enabled = true filter = apache-modsecurity logpath = /var/log/apache2/modsec_audit.log maxretry = 3 bantime = 86400

커스텀 규칙 예제

ModSecurity 커스텀 규칙

# SQL Injection 탐지 vi /etc/httpd/modsecurity.d/basic_rules.conf # SQL Injection 패턴 차단 SecRule ARGS "@detectSQLi" \ "id:1001,\ phase:2,\ deny,\ msg:'SQL Injection Attack Detected',\ logdata:'Matched Data: %{MATCHED_VAR} found within %{MATCHED_VAR_NAME}',\ tag:'attack-sqli',\ severity:CRITICAL" # Time-based SQL Injection 차단 (조건문과 지연 함수를 보내 진위를 판별) SecRule ARGS "@rx (?i:(sleep|waitfor|delay|benchmark)\s*\()" \ 정규식 설명:
(?i: … ) → 대소문자 구분 없이(case-insensitive) 매칭하라는 정규식 플래그.
(sleep|waitfor|delay|benchmark)\s*\( → sleep(, waitfor(, delay(, benchmark( 같은 함수 호출 패턴을 찾음 (\s*은 공백 허용). "id:1002,\ phase:2,\ deny,\ msg:'Time-based SQL Injection Attack',\ logdata:'Matched Data: %{MATCHED_VAR}',\ tag:'attack-sqli',\ severity:CRITICAL" # PostgreSQL sleep 함수 차단 SecRule ARGS "@rx (?i:pg_sleep\s*\()" \ "id:1003,\ phase:2,\ deny,\ msg:'PostgreSQL Sleep Function Detected',\ logdata:'Matched Data: %{MATCHED_VAR}',\ tag:'attack-sqli',\ severity:CRITICAL" # SQL 주석 패턴 차단 SecRule ARGS "@rx (?i:--\s*\/\*|\*\/|\/\*.*\*\/)" \ "id:1004,\ phase:2,\ deny,\ msg:'SQL Comment Pattern Detected',\ logdata:'Matched Data: %{MATCHED_VAR}',\ tag:'attack-sqli',\ severity:HIGH" # DECLARE/SET 구문 차단 SecRule ARGS "@rx (?i:declare\s*.*\s*char|set\s*.*\s*char)" \ "id:1005,\ phase:2,\ deny,\ msg:'SQL DECLARE/SET Statement Detected',\ logdata:'Matched Data: %{MATCHED_VAR}',\ tag:'attack-sqli',\ severity:CRITICAL" # 괄호를 이용한 SQL 구문 차단 SecRule ARGS "@rx \)\s*;\s*(SELECT|INSERT|UPDATE|DELETE|DROP)" \ "id:1006,\ phase:2,\ deny,\ msg:'SQL Statement After Closing Parenthesis',\ logdata:'Matched Data: %{MATCHED_VAR}',\ tag:'attack-sqli',\ severity:CRITICAL" # category_id는 숫자만 허용 SecRule ARGS:category_id "!@rx ^[0-9]+$" \ "id:1007,\ phase:2,\ deny,\ msg:'Invalid category_id format - only numbers allowed',\ logdata:'Invalid category_id: %{MATCHED_VAR}',\ tag:'input-validation',\ severity:HIGH" # 사용자 로그인 시 비밀번호를 로그에 남기지 않도록 추가 SecRule ARGS:passwd "@rx .*" "id:1008,phase:2,pass,nolog,sanitiseArg:passwd"

화이트 리스트

vi /etc/httpd/modsecurity.d/whitelist.conf SecRule REMOTE_ADDR "@ipMatch 192.168.0.1" \ "id:2001,\ phase:1,\ pass,\ ctl:ruleEngine=Off,\ msg:'Admin IP whitelist'"

적용 후 문제점 및 해결

❌ 문제: 오탐지

  • 정상적인 요청을 공격으로 오인하여 차단
  • 예: 쿠키에 포함된 문자열에서 공격 패턴을 탐지
  • 예: 비밀번호에 포함된 특수문자를 탐지

✅ 해결: 규칙 수정

  • 오탐지가 빈번한 규칙은 제거
  • 비밀번호 예외 처리
  • 점진적 규칙 적용 (DetectionOnly → On)
  • 지속적인 모니터링 필요

❌ 문제: 개인정보 이슈

  • 사용자의 비밀번호 입력을 로그에 기록

✅ 해결: 규칙 추가

  • 로그인 시 비밀번호 항목을 로그에 남기지 않도록

❌ 문제: CDN, 프록시를 이용할 경우

  • 원본이 아닌 proxy ip를 차단

⚠️ 해결: 원본 IP를 차단하도록 수정

  • 지속적으로 감옥을 확인해서 정상 사용자 또는 프록시 IP를 해방시켜주어야 함

모니터링 및 관리

🔍 ModSecurity 모니터링

# 통계 확인 awk 'match($0, /\[msg "([^"]+)"\]/, m) { counts[m[1]]++ } END { for (reason in counts) { print counts[reason], reason } }' /var/log/httpd/modsec_audit.log* | sort -nr # 실시간 모니터링 seclog='tail -f /var/log/httpd/modsec_audit.log'

# 모든 jail 상태 확인 fail2ban-client status # 차단된 IP 목록 fail2ban-client status apache-modsecurity # IP 차단 해제 fail2ban-client set apache-modsecurity unbanip 192.168.1.100 # 로그 모니터링 f2blog='tail -f /var/log/fail2ban.log'
