로그 관리의 문제점 및 필요성

🔴 주요 문제점

  • T사에서 초 단위로 주문내역 페이지 조회 요청
  • ->개인정보 조회기록이 기하급수적으로 증가
  • ->대용량 로그 데이터로 인해 조회, 삽입 성능 저하

✅ 개선 방향

  • 개인정보 조회기록은 수정되는 경우가 별로 없고,
  • 주 용도는 보관 및 조회
  • RDB보다 적합한 저장소로의 변경 필요

Elasticsearch 인덱스 및 저장 구조

📊 데이터 구조 계층

Index (인덱스)
RDB의 Database에 해당
↳ Document (문서)
RDB의 Row에 해당
JSON 형태로 저장

🔍 인덱스 구조 예시

// 인덱스 이름
logs-app-2024.11.04

// Document 예시
{
  "@timestamp": "2024-11-04T10:30:00",
  "level": "ERROR",
  "message": "Connection timeout",
  "user_id": "user123",
  "server": "web-01",
  "response_time": 3500
}

💡 핵심 개념

🖥️ Node
  • ES 클러스터의 단일 서버
  • 데이터 저장 및 검색 수행
  • 역할별 노드 구성 가능
🎯 Shard (Primary shard)
  • 인덱스 데이터를 여러 조각으로 나눈 단위
  • 수평적 확장 가능
  • 병렬 검색으로 성능 향상
🔄 Replica (Replica shard)
  • 샤드의 복사본
  • 고가용성 보장
  • 읽기 처리량 증가
🔗 노드 역할 (Node Roles)
Master Node: 클러스터 관리 및 메타데이터 관리
Data Node: 데이터 저장 및 검색 수행
Ingest Node: 데이터 전처리 및 변환
Coordinating Node: 요청 라우팅 및 결과 집계
🔗 역 인덱스 구조

문서에 포함된 단어를 키로 하고, 그 단어가 등장한 문서 목록을 값으로 저장

즉, "단어 → 문서들" 형태로 색인해 검색 시 빠르게 해당 문서를 찾는다.

적용 중 문제점 및 해결

❌ 문제: m.log()의 구성 문제

  • filebeat는 1 라인 = 1 이벤트로 인식하므로
  • 날짜와 데이터가 두 줄로 구성되어 파싱 어려움
  • 예시:
    [2025-07-08 17:23:08] 127.0.0.1 : /sysop/user/user_list.jsp
    {"purpose":"이러닝 운영","memo":"","reg_date":"20250708172308"...}
  • 멀티라인 코덱 적용 시 로드 폭등 발생

✅ 해결: JSON 데이터 구조 개선

  • JSON 데이터 내부에 날짜 정보를 포함시켜 단일 라인으로 구성
